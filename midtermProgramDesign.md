<!-- ZooKeeperChallenge Design Document

The ZooKeeperChallenge program is designed to read animal data from two input text files, create animal objects, and generate a final zoo population report. The program is built around an object-oriented design, with an abstract Animal superclass and four subclasses: Hyena, Lion, Tiger, and Bear. The Animal class stores all shared data, including name, age, species, sex, color, weight, origin, birth date, arrival date, and ID. Each subclass inherits those shared properties and only adds species-specific behavior by overriding the getHabitatTitle() method so the final report can group animals by habitat.

The program has a few main components. One part reads animalNames.txt and stores names by species. Another part reads arrivingAnimals.txt, parses each line, and uses that information to build the correct animal subclass object. A report-writing component then takes the finished list of animals and writes them into zooPopulation.txt, grouped by habitat and followed by species totals and the overall zoo total.

The main data structures are maps and lists. A Map<String, List<String>> stores the available names for each species. A Map<String, Integer> keeps track of which name should be assigned next for each species. Another Map<String, Integer> tracks the ID numbers, and one more stores how many animals of each species were added. The actual animal objects are stored in a List<Animal>, which allows the program to treat all animals the same while still using subclass behavior when needed.

The interaction between components is straightforward. First, the program loads the names into memory. Next, it reads each arriving animal line, parses the text, assigns the next available name and ID, and creates the correct subclass object. That object is added to the main animal list, and the species count is updated. After all lines are processed, the report component loops through the stored animals, organizes them by habitat title, and writes the final formatted output file. This design keeps the code modular, reduces repetition, and makes it easier to expand later if more animal types need to be added. -->
